# qrep

> Browse, search, and execute files from remote GitHub repositories via the GitHub API and curl.

**Version:** 2.0.0

## Purpose

qrep solves the problem of quickly locating and optionally executing scripts stored in a remote GitHub repository without cloning it locally. It is useful for operations teams or developers who maintain a shared repository of runbooks, snippets, or automation scripts and want a single CLI to discover and run them on any machine.

## Synopsis

```
qrep <command> [<file>]
```

## Description

The script loads a layered YAML configuration from four candidate paths (/etc, XDG_CONFIG_HOME, ~/.qrep-config.yaml, and ./.qrep-config.yaml), deep-merging each into a set of compiled defaults. It communicates with the GitHub REST API over HTTPS using Net::HTTP to fetch the full recursive git tree of the configured repository and branch. The 'find' sub-command filters tree blobs by basename, 'show' dumps the merged config as JSON, and 'exec' locates a file then pipes its raw content through curl into a configurable shell via Kernel#exec, with an optional interactive confirmation prompt.

## Notable Qualities

The use of Marshal.dump/load for deep-cloning the DEFAULTS hash is a classic Ruby trick that avoids accidental mutation of the frozen constant. The 'exec' command replaces the Ruby process entirely with a shell pipeline (curl | bash), meaning no Ruby code runs after that point, and the exit code is that of the pipeline. Configuration layering mimics the precedence model of tools like git-config, where project-local files override user and system settings.

## Bugs / Limitations

No GitHub authentication token support, so the script is subject to the unauthenticated GitHub API rate limit (60 requests/hour). The raw URL constructed for 'exec' is not shell-escaped, which could be exploited if file paths in the repository contain shell metacharacters. The 'exec' command replaces the process so no post-execution cleanup or error handling is possible from the script itself. If neither org nor repo is configured the script will produce an unhelpful GitHub API error rather than a clear validation message.
